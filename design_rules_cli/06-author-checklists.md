# Author Checklists

## Concept Checklist

- Is the player role clear?
- Is the end goal explicit?
- Does the theme naturally produce records and evidence?
- Can the game be described in one sentence?

## Puzzle Checklist

- Is there one obvious starting point?
- Are there 2 to 4 meaningful pivots?
- Does the final answer require multiple confirming clues?
- Is there at least one fair false lead?
- Can the player get permanently stuck?

## Data Checklist

- Do all references resolve?
- Are names consistent across files?
- Are schemas easy to parse in terminal output?
- Is the noise structured?
- Are there enough filler records to make search matter?

## Writing Checklist

- Does the intro establish role, mission, and first step quickly?
- Are hints written as operational guidance?
- Are key witness texts distinct enough to remember?
- Is there too much lore before play begins?

## Engineering Checklist

- Can the game be solved with standard shell tools?
- Are filenames shell-friendly?
- Are any files or formats platform-fragile?
- If tooling exists, does it validate the real clue path?
- Is the answer verification separate from ordinary browsing?

## Playtest Checklist

- Did a fresh player solve it without author rescue?
- Did they use the intended commands?
- Did they misunderstand any file family?
- Did any clue overperform or underperform?
- Did any shortcut collapse the whole puzzle?

## Release Checklist

- README present
- instructions present
- hints present
- answer verification present
- license present
- docs present

## Final Gate

Do not ship until you can say:

- the puzzle is solvable
- the puzzle is fair
- the puzzle is interesting
- the filesystem is part of the fun
