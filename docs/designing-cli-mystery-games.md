# Authoring Guide For CLI Mystery Games

## Purpose

This guide explains how to use the scaffold generated by `cli-mysterypack` to build
an actual mystery game that is playable in a terminal.

It is not package API documentation. It is design and production guidance for authors.

## Start From The Generated Scaffold

After running `init`, do not begin by expanding filler files at random. Work in this
order:

1. define the hidden truth
2. define the final answer format
3. design the clue graph
4. define data schemas
5. author the opening incident file
6. add supporting evidence families
7. write progressive hints
8. validate and playtest

This keeps the mystery solvable and coherent.

## What The Scaffold Gives You

The starter pack gives you:

- a project skeleton
- placeholder files for the first evidence families
- a story bible template
- a clue graph template
- starter hints
- a minimal validation pass

It does not give you:

- a complete puzzle
- balanced clue pacing
- data generation logic
- proof that the solution is unique

Those remain author responsibilities.

## Core Design Principle

The filesystem should act as both:

- the content store
- the navigation surface

Players should make progress by combining normal shell operations such as `cd`, `cat`,
`grep`, `head`, and `tail`.

## Use The Design Folder Properly

Before you heavily edit `game/`, fill out the authoring artifacts in `design/`.

### `design/story_bible.md`

Use this file to record:

- what happened
- who is responsible
- motive
- method
- timeline
- false leads
- confirming evidence

### `design/clue_graph.md`

Use this file to define:

- the starting clue
- the first pivot
- intermediate narrowing steps
- corroborating evidence
- the final confirmation step

If the clue graph is vague, the file set will drift into noise.

## Recommended Evidence Families

The scaffold defaults to several useful evidence families:

- `game/incident`: opening narrative and clue markers
- `game/people`: index of named entities and lookup targets
- `game/locations/`: address or room lookups
- `game/interviews/`: witness or suspect testimony
- `game/memberships/`: rosters for narrowing candidates
- `game/logs/`: timestamps, access records, or activity traces
- `game/registry/`: structured records such as vehicles, shipments, or licenses

You do not need all of them, but each one should answer a distinct class of question.

## Writing The Opening Incident File

`game/incident` should do three jobs:

1. establish the mystery
2. teach the player where useful clues live
3. expose the first real pivots

The built-in validator assumes the incident file contains the configured clue marker.
Keep that marker visible until you intentionally move to a more advanced design.

## Designing Good Clues

Strong clues:

- point toward another evidence family
- narrow the search space without solving the puzzle outright
- create a reason to use shell commands
- combine cleanly with later evidence

Weak clues:

- name the culprit too directly
- depend on guesswork
- only restate what the player already knows
- lead into dead ends with no feedback

## Writing Hints

Use the generated `hints/` files as a progression ladder.

Good hint progression:

- hint 1: identify the relevant file family
- hint 2: suggest the command family
- hint 3: explain how to combine evidence
- hint 4: reveal a near-direct pivot

Hints should reduce frustration without collapsing the puzzle.

## Custom Config Guidance

Use `--config` when the default assumptions are wrong for your game.

Useful config changes include:

- changing `display_title`
- changing `theme`
- changing `player_role`
- changing `clue_marker`
- adjusting generated folders for your evidence model

If you change the clue marker, make sure your authored `game/incident` still contains it
or `validate` will fail.

## Validation Workflow

Use validation throughout authoring, not only at the end:

```bash
python -m cli_mystery_starter validate .
```

This catches:

- missing required files
- invalid JSON in `mystery_config.json`
- missing clue marker in `game/incident`
- an empty `game/people`

It does not catch puzzle-quality issues, so pair it with manual review and playtesting.

## Playtesting Checklist

Before shipping a mystery built from this scaffold, verify:

- the intended solution is inferable
- no single search returns the answer immediately
- every important clue has a follow-up target
- the hints are progressive
- filenames and paths are shell-friendly
- the answer verification path works
- the story bible and final data still agree

## When To Add Code

Keep code secondary. Add scripts in `tools/` only when they reduce repetitive work, such
as:

- generating filler records
- building large rosters
- checking schema consistency
- confirming there is exactly one matching suspect
- producing the final encoded answer

If you can hand-author the puzzle cleanly, that is often the better first version.

## Recommended Maintainer Extensions

If you continue evolving this starter pack, the highest-value next additions are:

- stronger schema validation
- optional solution-consistency checks
- richer template variants by genre
- better example configs
- documented author workflows for beginner and advanced puzzles

## Bottom Line

Use the scaffold as structure, not as the design itself. The quality of the final game
still depends on authoring a clear hidden truth, a coherent clue graph, and evidence that
rewards command-line investigation.
